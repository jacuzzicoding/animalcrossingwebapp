# v0.9.1-beta — ACGCN Item Icons (Scoping Plan)

Status: scoping only. No scraping, wiring, or migrations until this plan is greenlit.

ACGCN is first because it's the smallest catalog (118 items) and the most stable
data source — getting the pipeline right on a small set lets the same approach
roll forward to ACWW/ACCF/ACNL/ACNH with low risk.

## 1. Inventory

Source of truth: `public/data/acgcn/*.json`.

| Category   | Count | Sample item names                                      |
|------------|-------|--------------------------------------------------------|
| `fish`     | 40    | Sea Bass, Brook Trout, Crucian Carp, Carp, Koi         |
| `bugs`     | 40    | Common Butterfly, Tiger Swallowtail Butterfly, Brown Cicada |
| `fossils`  | 25    | T. Rex (Skull/Torso/Tail), Pteranodon (Left Wing), Plesiosaurus (Neck) |
| `art`      | 13    | Academic Painting, Amazing Painting, Basic Painting     |
| **Total**  | **118** |                                                      |

Notes:
- Fossils are stored per-piece (e.g. `t-rex-skull`, `t-rex-torso`, `t-rex-tail`) and that's the granularity the museum tracks. Icons must be per-piece, not per-species.
- ACGCN has **no sea creatures** (sea creatures debuted in ACNH). The category JSON is absent and `sea_creatures.json` only exists under `public/data/acnh/`.
- Item schemas have no `imageUrl` field — icons will be resolved by id, not by an embedded URL.

## 2. Source evaluation

### Nookipedia REST API — *recommended primary*
- `https://api.nookipedia.com` with documented endpoints for `/nh/fish`, `/nh/bugs`, `/nh/sea`, `/nh/fossils`, `/nh/art`, plus generic `/villagers`. Each item response includes an `image_url` pointing at a hosted PNG.
- Requires a free API key (request via Nookipedia Discord) and a `User-Agent` header. No hard rate limit published, but politeness expected.
- Content is CC BY-SA 4.0 (Nookipedia text+images on the site itself; image rights ultimately Nintendo's — fan-content tolerance, with attribution required).
- **Coverage caveat (open question):** Nookipedia's API endpoints are NH-first. Older games (ACGCN/ACWW/ACCF/ACNL) live as wiki pages but may not all be queryable as structured JSON. For *icons specifically* the situation is friendlier than for stats/timing data: Nookipedia tends to have an article + sprite/render for each ACGCN item, accessible via the MediaWiki page-info / image API even when the structured `/gcn/...` endpoint doesn't exist. **Need to confirm before scraping kicks off** — see Open Questions §7.

### Fandom AC wiki (`animalcrossingwiki.fandom.com`) — *fallback*
- Per-item pages with infobox renders. Scrape via the MediaWiki API (`api.php?action=parse`) rather than HTML — no auth, but Fandom rate-limits aggressively and serves a lot of JS/ads in raw HTML.
- CC BY-SA 3.0. Image quality varies; many ACGCN sprites are ripped at native resolution (~32×32) with inconsistent backgrounds.
- Risk: Fandom has a history of breaking scrapers and pushing community wikis to fork. Treat as a backstop, not a primary.

### Community datasets surveyed
- Several GitHub repos host extracted **NH** asset packs (e.g. `ACNHAPI`, `acnhcdn.com`) — none cover ACGCN at the catalog level I could verify.
- The sister project (`../AnimalCrossingGCN-Tracker`) is a Swift app — worth a quick look during implementation to see if it bundles any ACGCN sprites that could be reused (likely no, but free if yes).

### Pick: **Nookipedia primary, Fandom fallback**
Rationale: Nookipedia is the closest thing the AC fandom has to an authoritative, attributable, machine-readable archive, and a single API key + polite scraper is far less brittle than HTML-scraping Fandom. Falling back to Fandom for any holes keeps us off Nintendo image hosts entirely.

## 3. Storage and naming

Layout:
```
public/icons/acgcn/
  fish/<id>.png
  bugs/<id>.png
  fossils/<id>.png
  art/<id>.png
```

- `id` = the existing item id from the JSON (already kebab-case, already stable, already what the store keys on). No re-slugging step, no second source of truth.
- Worked examples:
  - Fish "Pale Chub" → `pale-chub` → `public/icons/acgcn/fish/pale-chub.png`
  - Bug "Tiger Swallowtail Butterfly" → `tiger-swallowtail-butterfly` → `…/bugs/tiger-swallowtail-butterfly.png`
  - Fossil "T. Rex" Skull → `t-rex-skull` → `…/fossils/t-rex-skull.png`
  - Fossil "Pteranodon" Left Wing → `pteranodon-left-wing` → `…/fossils/pteranodon-left-wing.png`
  - Art "Academic Painting" → `academic-painting` → `…/art/academic-painting.png`
- Format: **PNG**, normalized to **64×64** (2× retina @ 32 logical px). Source images are mostly already small sprites; upscale only via integer nearest-neighbor if needed.
- Size cap: 20 KB per icon (118 × 20 KB ≈ 2.4 MB total). Almost certainly far less in practice.
- Serving: `/public` static — fetched by URL, never imported. Keeps the JS bundle untouched and lets the browser cache them naturally.
- Missing-icon fallback: a single shared placeholder at `public/icons/placeholder.png` (or render the existing category icon at the same size). Component decides via `onError` on `<img>`.

## 4. UI integration points

| Surface                        | File                                          | Proposed treatment                                     |
|--------------------------------|-----------------------------------------------|--------------------------------------------------------|
| Item row in a category tab     | `src/components/CollectibleRow.tsx`           | 32×32 icon left of the name, before the meta chips     |
| Inline expand panel            | `src/components/ItemExpandPanel.tsx`          | 64×64 icon top-left of the panel header                |
| Global search results          | `src/components/search/GlobalSearchDropdown.tsx` | 24×24 icon on each result row                         |
| Home tab — recent / leaving    | `src/components/HomeTab.tsx`                  | 24×24 icon next to item name in each strip             |
| Stats tab                      | `src/components/StatsTab.tsx`                 | No item-level rendering at present (charts/cards only) — skip |

Behavior on missing icon: render the placeholder; never collapse layout. Sizing must be reserved before the image loads to prevent CLS.

A thin shared component (`<ItemIcon item gameId size />`) is the right shape — single place that knows the URL convention and the fallback. Not building it in this PR; flagging it for PR (b).

## 6. Work plan

Three small PRs to keep review surface tight:

**PR (a) — Scraper + raw assets (~2 days)**
- `scripts/fetch-icons.ts` (Node, run locally, not at build time).
- Reads `public/data/acgcn/*.json`, queries Nookipedia per category, writes to `public/icons/acgcn/<cat>/<id>.png`.
- Logs missing items; falls back to Fandom MediaWiki API for any holes.
- Commits the resulting 118 PNGs + a `scripts/README.md` documenting how to re-run.
- File count: ~3 source files + 118 binaries.

**PR (b) — UI wiring + fallback (~1 day)**
- Add `<ItemIcon>` shared component with `onError` placeholder.
- Slot into the four surfaces in §4. Reserve dimensions to prevent layout shift.
- One small Vitest covering URL construction and fallback behavior.

**PR (c) — Release prep (~½ day)**
- `/credits` route with attribution copy.
- CHANGELOG entry under v0.9.1-beta.
- CLAUDE.md: note `public/icons/<gameId>/...` convention and the shared `<ItemIcon>` component in the file map.
- Tag `v0.9.1-beta` after merge to `main`.

This split keeps the binary-heavy commit isolated from the React changes (easier review, easier revert), and pushes attribution + docs to the very end where they're least likely to need rework.

## 7. Open questions

1. **Nookipedia coverage for ACGCN.** Highest-risk item. Need to spike the API + MediaWiki page-info endpoint against ~10 representative ACGCN items (1 art, 2 fossils, 3 fish, 3 bugs, including weird-name cases) before committing to it as primary. If coverage is <90% I'd rather flip the order and use Fandom primary.
2. **API key handling.** The Nookipedia key shouldn't live in the repo. Plan: read from `process.env.NOOKIPEDIA_API_KEY` in the script; document setup in `scripts/README.md`. Confirm that's acceptable before PR (a).
3. **Image normalization.** Nookipedia images for older games are sometimes ripped at native sprite resolution and sometimes at higher-res render quality. Acceptable to ship mixed-resolution icons in v0.9.1-beta and tighten consistency later, or do we want a normalization pass (transparent padding to a square canvas, consistent DPI) up front?
4. **Per-piece fossil icons.** Open whether Nookipedia has separate images per fossil piece (skull / torso / tail) or only per species. If only per species, all three pieces share one icon — acceptable degradation, but worth flagging before users see it.
5. **License compatibility.** The repo has no LICENSE file currently visible at top-level — should one be added (or at least a NOTICE) before redistributing CC BY-SA 4.0 assets? Probably yes; small task, easy to fold into PR (c).
