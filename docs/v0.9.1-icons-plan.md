# v0.9.1-beta — ACGCN Item Icons (Scoping Plan)

Status: scoping only. No scraping, wiring, or migrations until this plan is greenlit.

ACGCN is first because it's the smallest catalog (118 items) and the most stable
data source — getting the pipeline right on a small set lets the same approach
roll forward to ACWW/ACCF/ACNL/ACNH with low risk.

## 1. Inventory

Source of truth: `public/data/acgcn/*.json`.

| Category   | Count | Sample item names                                      |
|------------|-------|--------------------------------------------------------|
| `fish`     | 40    | Sea Bass, Brook Trout, Crucian Carp, Carp, Koi         |
| `bugs`     | 40    | Common Butterfly, Tiger Swallowtail Butterfly, Brown Cicada |
| `fossils`  | 25    | T. Rex (Skull/Torso/Tail), Pteranodon (Left Wing), Plesiosaurus (Neck) |
| `art`      | 13    | Academic Painting, Amazing Painting, Basic Painting     |
| **Total**  | **118** |                                                      |

Notes:
- Fossils are stored per-piece (e.g. `t-rex-skull`, `t-rex-torso`, `t-rex-tail`) and that's the granularity the museum tracks. Icons must be per-piece, not per-species.
- ACGCN has **no sea creatures** (sea creatures debuted in ACNH). The category JSON is absent and `sea_creatures.json` only exists under `public/data/acnh/`.
- Item schemas have no `imageUrl` field — icons will be resolved by id, not by an embedded URL.

## 2. Source evaluation

### Fandom AC wiki MediaWiki API — *primary*
- Endpoint: `https://animalcrossing.fandom.com/api.php`. Use `action=query&prop=pageimages|images` to resolve a page's lead image, then `prop=imageinfo&iiprop=url` to get the direct file URL. No auth, no key.
- Per-item pages exist for the full ACGCN catalog (every fish/bug/fossil/art piece has its own article on the AC Fandom wiki).
- Content is CC BY-SA 3.0. Attribution required (link back to the Fandom wiki); underlying IP is Nintendo's, surfaced as a NOTICE.
- Risk: Fandom rate-limits aggressively. Scraper runs locally with a polite delay (≥500 ms between requests) and a descriptive `User-Agent`. A full ACGCN run is 118 calls — single-digit minutes worst case.
- Image quality varies; ACGCN sprites are often native-resolution (~32–64 px) with transparent backgrounds. Acceptable for v0.9.1 per §3.

### HTML scrape — *last-resort fallback*
- For any item the MediaWiki API can't resolve, fetch the article HTML and parse the infobox `<img>`. Slower, more brittle, but requires no extra infrastructure.
- Same license + attribution surface as the API path (still Fandom content).

### Nookipedia REST API — *off the table*
- Decided not to use it. Requires a free API key obtained out-of-band (Discord request), and there's no plan to maintain that credential. Default for all v0.9.x icon releases is Fandom-primary.

### Community datasets surveyed
- Several GitHub repos host extracted **NH** asset packs (e.g. `ACNHAPI`, `acnhcdn.com`) — none cover ACGCN at the catalog level worth pulling in.
- The sister project (`../AnimalCrossingGCN-Tracker`) is a Swift app — quick check during implementation in case it bundles reusable ACGCN sprites (likely no, but free if yes).

### Pick: **Fandom MediaWiki API primary, Fandom HTML scrape fallback**
Rationale: zero-credential, zero-cost, single source = single attribution surface. One source means one license footer and one place to point users for image rights questions. Brittleness is mitigated by the polite-delay scraper and the small catalog size.

## 3. Storage and naming

Layout:
```
public/icons/acgcn/
  fish/<id>.png
  bugs/<id>.png
  fossils/<id>.png
  art/<id>.png
```

- `id` = the existing item id from the JSON (already kebab-case, already stable, already what the store keys on). No re-slugging step, no second source of truth.
- Worked examples:
  - Fish "Pale Chub" → `pale-chub` → `public/icons/acgcn/fish/pale-chub.png`
  - Bug "Tiger Swallowtail Butterfly" → `tiger-swallowtail-butterfly` → `…/bugs/tiger-swallowtail-butterfly.png`
  - Fossil "T. Rex" Skull → `t-rex-skull` → `…/fossils/t-rex-skull.png`
  - Fossil "Pteranodon" Left Wing → `pteranodon-left-wing` → `…/fossils/pteranodon-left-wing.png`
  - Art "Academic Painting" → `academic-painting` → `…/art/academic-painting.png`
- Format: **PNG**, normalized to **64×64** (2× retina @ 32 logical px). Source images are mostly already small sprites; upscale only via integer nearest-neighbor if needed.
- Size cap: 20 KB per icon (118 × 20 KB ≈ 2.4 MB total). Almost certainly far less in practice.
- Serving: `/public` static — fetched by URL, never imported. Keeps the JS bundle untouched and lets the browser cache them naturally.
- Missing-icon fallback: a single shared placeholder at `public/icons/placeholder.png` (or render the existing category icon at the same size). Component decides via `onError` on `<img>`.

## 4. UI integration points

| Surface                        | File                                          | Proposed treatment                                     |
|--------------------------------|-----------------------------------------------|--------------------------------------------------------|
| Item row in a category tab     | `src/components/CollectibleRow.tsx`           | 32×32 icon left of the name, before the meta chips     |
| Inline expand panel            | `src/components/ItemExpandPanel.tsx`          | 64×64 icon top-left of the panel header                |
| Global search results          | `src/components/search/GlobalSearchDropdown.tsx` | 24×24 icon on each result row                         |
| Home tab — recent / leaving    | `src/components/HomeTab.tsx`                  | 24×24 icon next to item name in each strip             |
| Stats tab                      | `src/components/StatsTab.tsx`                 | No item-level rendering at present (charts/cards only) — skip |

Behavior on missing icon: render the placeholder; never collapse layout. Sizing must be reserved before the image loads to prevent CLS.

A thin shared component (`<ItemIcon item gameId size />`) is the right shape — single place that knows the URL convention and the fallback. Not building it in this PR; flagging it for PR (b).

## 5. Attribution

Two surfaces, populated in PR (c):

- **`NOTICE` at the repo root** — plain-text statement that item icons are sourced from the [Animal Crossing Fandom wiki](https://animalcrossing.fandom.com), used under CC BY-SA 3.0, with underlying IP held by Nintendo.
- **In-app `/credits` route** — same attribution copy, link to the Fandom wiki, link to the CC BY-SA 3.0 deed, and the Nintendo IP notice. Reachable from a footer link in the main shell. No per-icon attribution overlay.

## 6. Work plan

Three small PRs to keep review surface tight:

**PR (a) — Scraper + raw assets (~2 days)**
- `scripts/fetch-icons.ts` (Node, run locally, not at build time).
- Reads `public/data/acgcn/*.json`, queries the Fandom MediaWiki API per item, downloads the lead image, writes to `public/icons/acgcn/<cat>/<id>.png`.
- Polite delay (≥500 ms) between requests, descriptive `User-Agent`.
- Logs misses; for any item the API path can't resolve, falls back to HTML-scraping the article infobox.
- Commits the resulting PNGs + a `scripts/README.md` documenting how to re-run.
- File count: ~3 source files + ~118 binaries.

**PR (b) — UI wiring + fallback (~1 day)**
- Add `<ItemIcon>` shared component with `onError` placeholder.
- Slot into the four surfaces in §4. Reserve dimensions to prevent layout shift.
- One small Vitest covering URL construction and fallback behavior.

**PR (c) — Release prep (~½ day)**
- `NOTICE` file at repo root + `/credits` route with matching attribution copy (Fandom wiki, CC BY-SA 3.0, Nintendo IP).
- CHANGELOG entry under v0.9.1-beta — call out the per-piece-fossil degradation if the Fandom source only carries per-species images.
- CLAUDE.md: note `public/icons/<gameId>/...` convention and the shared `<ItemIcon>` component in the file map.
- Tag `v0.9.1-beta` after merge to `main`.

This split keeps the binary-heavy commit isolated from the React changes (easier review, easier revert), and pushes attribution + docs to the very end where they're least likely to need rework.
