Status (updated 2026-05-04): PR (a) merged (#86) — scraper, resolver, and the full ACGCN icon set committed. PR (b) merged (#89) — `<ItemIcon>` component + UI wiring shipped (rows, expand panels, global search, home shelves) plus the data-driven manifest probe. PR (c) in flight on `release/v0.9.1-beta` — version bumps, CHANGELOG, `/credits` route, MIT LICENSE + NOTICE. Sections below are preserved for context; where the shipped implementation diverges from the original scoping, the divergence is called out inline.

ACGCN is first because it's the smallest catalog (118 items) and the most stable
data source — getting the pipeline right on a small set lets the same approach
roll forward to ACWW/ACCF/ACNL/ACNH with low risk.

## 2. Source evaluation

### Fandom AC wiki MediaWiki API — *primary*
- Endpoint: `https://animalcrossing.fandom.com/api.php`. Use `action=query&prop=pageimages|images` to resolve a page's lead image, then `prop=imageinfo&iiprop=url` to get the direct file URL. No auth, no key.
- Per-item pages exist for the full ACGCN catalog (every fish/bug/fossil/art piece has its own article on the AC Fandom wiki).
- Content is CC BY-SA 3.0. Attribution required (link back to the Fandom wiki); underlying IP is Nintendo's, surfaced as a NOTICE.
- Risk: Fandom rate-limits aggressively. Scraper runs locally with a polite delay (≥500 ms between requests) and a descriptive `User-Agent`. A full ACGCN run is 118 calls — single-digit minutes worst case.
- Image quality varies; ACGCN sprites are often native-resolution (~32–64 px) with transparent backgrounds. Acceptable for v0.9.1 per §3.

### HTML scrape — *last-resort fallback*
- For any item the MediaWiki API can't resolve, fetch the article HTML and parse the infobox `<img>`. Slower, more brittle, but requires no extra infrastructure.
- Same license + attribution surface as the API path (still Fandom content).

### Nookipedia REST API — *off the table*
- Decided not to use it. Requires a free API key obtained out-of-band (Discord request), and there's no plan to maintain that credential. Default for all v0.9.x icon releases is Fandom-primary.

### Community datasets surveyed
- Several GitHub repos host extracted **NH** asset packs (e.g. `ACNHAPI`, `acnhcdn.com`) — none cover ACGCN at the catalog level worth pulling in.
- The sister project (`../AnimalCrossingGCN-Tracker`) is a Swift app — quick check during implementation in case it bundles reusable ACGCN sprites (likely no, but free if yes).

### Pick: **Fandom MediaWiki API primary, Fandom HTML scrape fallback**
Rationale: zero-credential, zero-cost, single source = single attribution surface. One source means one license footer and one place to point users for image rights questions. Brittleness is mitigated by the polite-delay scraper and the small catalog size.

## 3. Storage and naming

Layout:
```
public/icons/acgcn/
  fish/<id>.<ext>
  bugs/<id>.<ext>
  fossils/<id>.<ext>
  art/<id>.<ext>
```

> **Shipped (PR #86):** source extension is preserved per file — Fandom serves a mix of `.png` and `.jpg` (with `.jpeg` normalized to `.jpg`), so the on-disk layout reflects whatever the wiki returned. Matching the source format avoids re-encoding artifacts and keeps PR (a) simple. A `manifest.json` per game maps `id → filename` for the consuming UI.

- `id` = the existing item id from the JSON (already kebab-case, already stable, already what the store keys on). No re-slugging step, no second source of truth.
- Worked examples:
  - Fish "Pale Chub" → `pale-chub` → `public/icons/acgcn/fish/pale-chub.png`
  - Bug "Tiger Swallowtail Butterfly" → `tiger-swallowtail-butterfly` → `…/bugs/tiger-swallowtail-butterfly.png`
  - Fossil "T. Rex" Skull → `t-rex-skull` → `…/fossils/t-rex-skull.png`
  - Fossil "Pteranodon" Left Wing → `pteranodon-left-wing` → `…/fossils/pteranodon-left-wing.png`
  - Art "Academic Painting" → `academic-painting` → `…/art/academic-painting.png`
- Format: **shipped as-fetched** for v0.9.1-beta — source-resolution PNG/JPG, source extension preserved, no normalization pass. The original scoping called for normalization to 64×64 PNG with a 20 KB per-icon cap; that was descoped in PR #86 to keep the binary commit minimal and avoid touching pixels we don't need to touch. A normalization pass remains an option for a later polish release if visual consistency or bundle size becomes a problem in practice.
- No hard size cap enforced — the full ACGCN run lands around 6 MB for 118 items, well under what would warrant pre-shipping resampling.
- Serving: `/public` static — fetched by URL, never imported. Keeps the JS bundle untouched and lets the browser cache them naturally.
- Missing-icon fallback: a single shared placeholder at `public/icons/placeholder.png` (or render the existing category icon at the same size). Component decides via `onError` on `<img>`.

## 6. Work plan

Three small PRs to keep review surface tight:

**PR (a) — Scraper + raw assets (~2 days)**
- `scripts/fetch-icons.ts` (Node, run locally, not at build time).
- Reads `public/data/acgcn/*.json`, queries the Fandom MediaWiki API per item, downloads the lead image, writes to `public/icons/acgcn/<cat>/<id>.png`.
- Polite delay (≥500 ms) between requests, descriptive `User-Agent`.
- Logs misses; for any item the API path can't resolve, falls back to HTML-scraping the article infobox.
- Commits the resulting PNGs + a `scripts/README.md` documenting how to re-run.
- File count: ~3 source files + ~118 binaries.

**PR (b) — UI wiring + fallback (~1 day)**
- Add `<ItemIcon>` shared component with `onError` placeholder.
- Slot into the four surfaces in §4. Reserve dimensions to prevent layout shift.
- One small Vitest covering URL construction and fallback behavior.

**PR (c) — Release prep (~½ day)**
- `NOTICE` file at repo root + `/credits` route with matching attribution copy (Fandom wiki, CC BY-SA 3.0, Nintendo IP).
- CHANGELOG entry under v0.9.1-beta — call out the per-piece-fossil degradation if the Fandom source only carries per-species images.
- CLAUDE.md: note `public/icons/<gameId>/...` convention and the shared `<ItemIcon>` component in the file map.
- Tag `v0.9.1-beta` after merge to `main`.

This split keeps the binary-heavy commit isolated from the React changes (easier review, easier revert), and pushes attribution + docs to the very end where they're least likely to need rework.

**Coverage: 8/10 (80%) — below the 90% bar. Halting before PR (a).**

### Why the misses happened (and how to close the gap)

Both misses are *resolution* failures, not *content* failures: the wiki almost certainly has images for both items, just not at the page title we tried. Two complementary fixes worth scoping before resuming:

1. **Title-resolution layer in front of the API call.** Try, in order: (a) the bare item `name`, (b) `name (category)`, (c) for art only, the `basedOn` field (strip "by &lt;artist&gt;"), (d) MediaWiki `list=search` for the item name and use the top hit, (e) `prop=pageimages` against any redirect target. Step (d) alone likely solves both current misses.
2. **Article-HTML fallback.** For anything still unresolved, fetch the article HTML and pull the first infobox `<img>`. Fandom infoboxes are consistent enough that one selector covers nearly every page.

Both are small additions to the spike script and would let it re-run and validate before any binary commits.

### Recommendation

Re-spike with the title-resolution layer added; if that pushes coverage to ≥90%, proceed to PR (a) as planned. If even one item still misses after both fallbacks, flag it for hand-resolution (manual page lookup, paste the title back into a config map) before scraping the full 118.

## 9. Re-spike results (resolution chain v2)

Extended `scripts/spike-fandom-coverage.ts` with a five-step chain — (a) bare name, (b) `name (category)`, (c) art-only `basedOn` stripped of " by …", (d) `list=search` top hit with loose match, (e) HTML infobox fallback — and re-ran the same 10 items.

| Result   | Id                              | Step that won | URL returned |
|----------|---------------------------------|---------------|--------------|
| OK       | t-rex-skull                     | a:bare        | Tyrannosaurus_Rex.png |
| OK       | mammoth-skull                   | a:bare        | Mammoth.jpeg |
| OK       | pale-chub                       | a:bare        | Pale_chub_nh.png |
| OK       | sea-bass                        | a:bare        | Nh-sea-bass.jpg |
| OK       | koi                             | a:bare        | Koi_nh.png |
| OK       | common-butterfly                | a:bare        | NH-Common_butterfly.jpg |
| OK       | brown-cicada                    | a:bare        | Brown_Cicada_NH.png |
| OK       | amazing-painting                | a:bare        | The_Nightwatch.jpg |
| **FALSE POSITIVE** | academic-painting    | e:html        | `Flowery_Painting.jpg` — wrong painting; the HTML fallback grabbed a thumbnail from a related-paintings list rather than the actual Academic Painting (which is the Vitruvian Man) |
| MISS     | tiger-swallowtail-butterfly     | none          | None of the five steps resolved a page |

**Reported coverage: 9/10 (90%). Honest coverage after counting the false positive: 8/10 (80%). Halting again before PR (a).**

### What the v2 chain actually fixed and what it didn't

- **Steps (a)–(c) work as intended** — bare name + per-category disambig is sufficient for the eight clean cases, and (c) is a sound design even though it didn't fire here (see below).
- **(c) didn't help `academic-painting`** because the Fandom wiki has no page literally titled "Vitruvian Man" — only the painting's article in the AC context. Step (c) should be promoted from "pageimages on the stripped basedOn" to "**search** for the stripped basedOn, then pageimages on the loose-matched top hit." That likely closes academic-painting cleanly.
- **(e) HTML fallback is unsafe as written.** It blindly trusts whatever search returns and pulls *any* infobox image off the resulting page. For `academic-painting`, search landed on a "Paintings" listing page whose first infobox image is a different painting entirely. The HTML path needs to either (i) be removed in favor of a stricter API-only chain, or (ii) only run when we have high confidence the resolved title actually belongs to the item (e.g. exact title match, or the article's category includes "ACGCN" / the item category).
- **`tiger-swallowtail-butterfly` is a real naming-drift miss.** The bug is almost certainly indexed under a shorter/older name on Fandom; the current loose-match in step (d) didn't accept whatever search returned. Worth printing the rejected search candidates next round so we can see what it almost accepted.

### Recommended next iteration

Three small, contained fixes — happy to do them in another turn before any binary commits:

1. **Promote (c) to a search-then-resolve flow** for art using the stripped `basedOn`. Likely solves `academic-painting`.
2. **Tighten the loose-match in (d)** to require all whitespace-separated tokens of the item name appear in the candidate title (case-insensitive). And log the rejected candidates so misses are diagnosable instead of silent.
3. **Restrict or remove (e).** Either drop it entirely (cleaner, fewer false positives) or only invoke it when (a)/(b) returned a *real* page that just had no `pageimages.original` — never when we're guessing at a title from search.

If those three changes push real coverage to 10/10 (or 9/10 with a single hand-resolved override for the holdout), proceeding to the full scrape is safe. If `tiger-swallowtail-butterfly` still misses, a one-line override map (`{ 'tiger-swallowtail-butterfly': 'Tiger butterfly' }` or whatever the actual page is) gets us across the line without scope creep.

## 10. v3 resolver results — algorithmic ceiling reached

Implemented all three v3 changes:
- Promoted art `basedOn` (with " by …" stripped) to the **canonical lookup string** for art items, used by every step in the chain.
- Tightened search loose-match: top-5 hits accepted only if all whitespace tokens of the lookup string appear in the candidate title (case-insensitive). Rejected candidates logged for diagnosis.
- Restricted HTML fallback to fire **only** against a page that earlier returned `kind: 'no-image'` (page exists but has no `pageimages.original`). Never against guessed titles.

Results:

| Result | Id                              | Via      | Title resolved      | Notes |
|--------|---------------------------------|----------|---------------------|-------|
| OK     | t-rex-skull                     | a:bare   | Tyrannosaurus rex   | |
| OK     | mammoth-skull                   | a:bare   | Mammoth             | |
| OK     | pale-chub                       | a:bare   | Pale chub           | |
| OK     | sea-bass                        | a:bare   | Sea bass            | |
| OK     | koi                             | a:bare   | Koi                 | |
| OK     | common-butterfly                | a:bare   | Common butterfly    | |
| OK     | brown-cicada                    | a:bare   | Brown cicada        | |
| MISS   | academic-painting               | none     | —                   | basedOn lookup "Vitruvian Man" misses; search returns unrelated pages (Southern hemisphere, Clownfish/Gallery, Winter Solstice) |
| MISS   | tiger-swallowtail-butterfly     | none     | —                   | bare misses; search returns unrelated bug-list pages |
| MISS   | amazing-painting                | none     | —                   | **regressed from OK in v2.** basedOn lookup "The Night Watch" misses; bare "Amazing Painting" would have hit |

**Honest coverage: 7/10 (70%). Worse than v2 (8/10 honest). Algorithmic ceiling reached.**

### Key finding: `basedOn` is the wrong canonical source for art

The Fandom AC wiki documents the **in-game items**, not the real-world paintings. So pages exist at "Amazing Painting" but not at "The Night Watch". Promoting `basedOn` to canonical broke `amazing-painting` for the same reason it failed to fix `academic-painting` — the wiki simply doesn't index by real-artwork title.

The right shape for art lookup is either:
- **Try both, item name first**: bare `name` → bare `basedOn` (stripped) → search on either. This recovers `amazing-painting` and at least gives `academic-painting` a second shot.
- Or, more honest given what we just learned: **drop `basedOn` from the resolver entirely** and treat both art misses as resolver-ceiling cases bound for the overrides map.

I'd take the second option — the data says `basedOn` doesn't pull its weight, and adding it back as a "try also" step just pads the chain with low-yield calls.

### Honest stoppers

- `academic-painting` and `tiger-swallowtail-butterfly` are unresolvable algorithmically against this wiki without context the resolver doesn't have. The wiki almost certainly indexes them under different titles (likely "Academic painting" already worked → wait, the test showed bare "Academic Painting" missing in v3 because we passed the basedOn instead of name; need to re-test bare "Academic Painting" with the v3 chain to be sure).
- `amazing-painting` regressed purely because the `basedOn` promotion was a wrong call. Easily fixable, but the broader lesson is the resolver is at its algorithmic ceiling regardless.

### Recommendation

Stop spinning resolver rounds. Two-step path forward:

1. **Tiny corrective tweak**: drop `basedOn` from the canonical-lookup logic so art falls back to its v2 behavior (bare name → disambig → search). That alone restores `amazing-painting` and brings us back to 8/10.
2. **Add a small `OVERRIDES` map** in the (forthcoming) `scripts/fetch-icons.ts` for the algorithmic holdouts. Each entry is one line: `{ 'academic-painting': 'Academic Painting', 'tiger-swallowtail-butterfly': 'Tiger butterfly' }` (real titles to be confirmed by manual wiki lookup). Three or four entries, max, across the entire ACGCN catalog.

The overrides map is honest scope: it acknowledges that wiki-naming drift is a fact of the source and shouldn't be papered over with ever-fancier resolver heuristics that introduce false positives. It's also a portable pattern — the same map shape will absorb the inevitable handful of holdouts on ACWW/ACCF/ACNL/ACNH.

Awaiting direction on whether to land that path.
