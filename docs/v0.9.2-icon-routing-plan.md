# v0.9.2-beta — Cross-Game Icon Routing (Scoping Plan)

**Status:** scoping pass; nothing implemented. Sibling of `v0.9.1-icons-plan.md`.

v0.9.1 shipped per-game manifests at `public/icons/<gameId>/manifest.json`, each shaped as `{ [category]: { [id]: filename } }`. `<ItemIcon>` resolves a triple `(gameId, category, id)` against exactly one game's manifest. The catalog audit showed roughly **56% of naive scrape work is redundant** across games — the same fish/bug appears in 3-5 catalogs under (mostly) the same id. v0.9.2 introduces a routing layer so a single icon for `fish/sea-bass` can serve every game that has that item, while preserving native-game icons when they exist (hand-drawn art may differ subtly between games).

---

## 1. Manifest schema design

Three options surveyed:

- **A. Reverse-index `public/icons/index.json`.** A single generated file: `{ "<category>/<id>": ["acgcn", "acww", ...], ... }`. One fetch per session. Per-game manifests stay (they're still the source of `id → filename`).
- **B. Per-game manifests embed `cross_refs`.** Each per-game manifest grows a `{ [category]: { [id]: ["other-game", ...] } }` block. No new file, but every manifest must be regenerated in lockstep when ANY game gains an icon — sync risk grows quadratically.
- **C. Both A and B (index is authoritative, per-game keeps cross_refs for partial-load scenarios).** Maximum redundancy, maximum complexity, no real payoff.

**Decision: Option A.**

Rationale:
- Fetch count: A is one extra fetch per session (≈1-3 KB gzipped for ~400 entries). B is zero extra fetches but at the cost of bloating every per-game manifest. C is one extra fetch AND bloated manifests.
- Sync risk: A regenerates from per-game manifests via a single script — the index can never drift out of sync with what's actually on disk because it's derived. B requires N-way edits and is human-error-prone. C compounds B's problem.
- Resolver complexity: A gives the resolver an O(1) lookup `index["fish/sea-bass"] → ["acgcn", "acww"]`; B requires iterating cross_refs across N manifests; C is the union of both.
- Backward compat: A leaves per-game manifests untouched, so v0.9.1's `useManifestState(gameId)` keeps working unchanged. The new resolver is additive.

Generator: `scripts/generate-icon-index.ts` reads all per-game manifests and emits `public/icons/index.json`. Wire into the existing pre-commit / pre-build hook used by the v0.9.1 scrape pipeline.

**Schema:**
```jsonc
{
  "schemaVersion": 1,
  "generatedAt": "2026-05-05T00:00:00Z",
  "items": {
    "fish/sea-bass": ["acgcn", "acww", "accf", "acnl", "acnh"],
    "bugs/citrus-long-horned-beetle": ["acww", "accf", "acnl", "acnh"]
  },
  "aliases": {
    "fish/citrus-longhorn-beetle": "fish/citrus-long-horned-beetle"
  }
}
```

The `aliases` block lives alongside `items` so it ships in the same fetch (see §3).

## 2. Resolver chain

`<ItemIcon>` becomes:

1. **Alias normalize.** If `"<category>/<id>"` is in `aliases`, swap to canonical key.
2. **Native.** If `manifest[gameId][category][id]` exists, use `/icons/<gameId>/<category>/<filename>`. Highest priority — preserves v0.9.1 behavior, and per-game art may differ.
3. **Routed.** If not native, look up `index.items["<category>/<canonicalId>"]`. Walk the routing priority order and return the first hit's URL: `/icons/<otherGame>/<category>/<filename>`. Manifest of the routed game must already be loaded (or the routing layer triggers its lazy fetch).
4. **Placeholder.** No game has it → existing monogram fallback.

**Routing priority order.**

Three candidates:
- `[ACGCN, ACWW, ACCF, ACNL, ACNH]` — oldest-first.
- `[ACNH, ACNL, ACCF, ACWW, ACGCN]` — newest-first.
- Per-category override (e.g. fossils prefer ACNH because of pieces, fish prefer ACGCN, etc.).

**Decision: oldest-first `[ACGCN, ACWW, ACCF, ACNL, ACNH]`, configurable.**

Rationale:
- Hand-drawn icons (per `roadmap-to-v1.md`) target ACGCN first as creatures debut chronologically. Routing to ACGCN first means every game's view benefits from the canonical hand-drawn art the moment it lands.
- During the wiki-placeholder window (v0.9.x betas), wiki art quality is roughly comparable across games; oldest-first gives a stable, predictable answer.
- Configurable via a single `ROUTING_PRIORITY: GameId[]` constant (in `src/lib/iconRouting.ts`). Flipping to newest-first or per-category is one edit, no resolver rewrite.

## 3. Override map for rename pairs

The catalog audit surfaced these aliases. For each, canonical id is **the older-game spelling** (consistency with the oldest-first routing order, and most older-game ids are simpler/closer to the wiki primary spelling):

| Alias (newer/variant) | Canonical (older) | Category | Notes |
|---|---|---|---|
| `citrus-longhorn-beetle` | `citrus-long-horned-beetle` | bugs | hyphenation drift |
| `rajah-brookes-birdwing` | `rajah-brooke-birdwing` | bugs | possessive drift |
| `man-faced-stink-bug` | `manfaced-stink-bug` | bugs | spacing |
| `giant-water-bug` | `giant-waterbug` | bugs | spacing |
| `queen-alexandras-birdwing` | `queen-alexandra-birdwing` | bugs | possessive |
| `sabertooth-skull` | `sabretooth-skull` | fossils | regional spelling |
| `sabertooth-torso` | `sabretooth-torso` | fossils | (per-piece variants) |
| `pachycephalosaur-skull` | `pachycephalosaurus-skull` | fossils | abbreviated form |
| `pachycephalosaur-tail` | `pachycephalosaurus-tail` | fossils | |
| `peking-man` | `peking-man-skull` | fossils | NH adds piece suffix |

**Discovery pass — additional candidates to verify** (grep across `public/data/<game>/*.json` during implementation; flag genuine renames only, not retired species):

- `tiger-beetle` vs `tiger-beetles` — possible plural drift, verify
- `walking-leaf` vs `leaf-insect` — semantic match, possible rename, verify
- `dab` vs `olive-flounder` — likely distinct species, do NOT alias without confirmation
- `goliath-beetle` vs `goliath-beetles` — verify

The implementation PR should commit any newly confirmed pairs to the alias map and document rejections in the PR body.

**Override-map shape:** `Record<"<category>/<alias-id>", "<category>/<canonical-id>">` evaluated as step 1 of the resolver chain, BEFORE native lookup. This means a game's native id that happens to be an alias still gets normalized — intentional, so the index lookup is uniform. Per-game manifests keep their own native ids unchanged (no rewrite of v0.9.1 data).

**Canonical-id selection rule:** prefer the older-game spelling. If there's a tie (item only in two newer games) prefer the wiki-primary spelling. Document the chosen rule in `iconRouting.ts` so future additions are mechanical.

## 4. ACNH fossil schema drift

Older games: per-skeleton row with a `part` column (`t-rex` skeleton, parts `[skull, torso, tail]`).
ACNH: per-piece row (`t-rex-skull`, `t-rex-torso`, `t-rex-tail`).

Three options:
- **A. Skip fossils from routing.** Each game's fossil scrape stays independent.
- **B. Name-based join (lowercased name as join key for fossils only).** Recovers more reuse, more code.
- **C. Hand-curated fossil-piece-to-skeleton map.** Most accurate, slowest to populate.

**Decision: Option A (skip fossil routing in v0.9.2).**

Rationale:
- The roadmap (`docs/roadmap-to-v1.md`) specifies fossils get a single generic placeholder for v1.0 hand-drawn delivery. Cross-game fossil routing is solving a problem that mostly evaporates by launch.
- Wiki-placeholder period (v0.9.x betas) still benefits from per-game native fossil icons; users see SOMETHING, just not routed art.
- Schema-drift mapping (B or C) is meaningful complexity that pays off only for ~3 months of beta. Defer; revisit only if fossil coverage gaps become a user-visible issue.

Implementation: `iconRouting.ts` exports `ROUTABLE_CATEGORIES: CategoryId[] = ['fish', 'bugs', 'art', 'sea_creatures']` (note: art is routable; ACNH/ACNL share painting ids cleanly per audit). Resolver short-circuits to native-only for `fossils`.

## 5. Audit script — `scripts/audit-icon-coverage.ts`

Spec:
- **Inputs:** all per-game manifests + all per-game data catalogs (`public/data/<gameId>/*.json`) + the alias map + `ROUTING_PRIORITY` + `ROUTABLE_CATEGORIES`.
- **Per item, per game, classify as:**
  - `native` — covered by this game's own manifest
  - `routed:<sourceGame>` — covered by routing from another game
  - `uncovered` — neither native nor routed
- **Output:** `docs/v0.9.2-icon-coverage-audit.md`.

**Output format:**

```markdown
# Icon Coverage Audit — generated <ISO date>

## Summary
| Game  | Catalog | Native | Routed | Uncovered | Effective coverage |
| ACGCN | 118     |  118   |   0    |     0     | 100% |
| ACWW  | 164     |   12   |  140   |    12     |  93% |
| ...

## Routing source breakdown
| Source game | Items routed FROM this game |
| ACGCN | 140 |
| ...

## Per-game uncovered (gap lists for v0.9.4-v0.9.7 scoping)

### ACWW (12 uncovered)
- bugs/<id>
- fish/<id>
...
```

The per-game uncovered list is the deliverable consumed by v0.9.4-v0.9.7 release scoping (each release scrapes only what its game's gap list says).

CLI: `npm run audit:icons` → writes the doc and prints the summary table.

## 6. Component / hook changes

New module `src/lib/iconRouting.ts`:
```ts
export const ROUTING_PRIORITY: GameId[] = ['ACGCN', 'ACWW', 'ACCF', 'ACNL', 'ACNH'];
export const ROUTABLE_CATEGORIES: CategoryId[] = ['fish', 'bugs', 'art', 'sea_creatures'];

export type ResolvedIcon =
  | { kind: 'native'; gameId: GameId; url: string }
  | { kind: 'routed'; sourceGameId: GameId; url: string }
  | { kind: 'placeholder' };

export function resolveIcon(
  requested: { gameId: GameId; category: CategoryId; id: string },
  manifests: Record<GameId, ManifestState>,
  index: IconIndex,
): ResolvedIcon;
```

New hook `useIconIndex()` (module-level cache, single fetch, shared via `useSyncExternalStore`). Same shape as `useManifestState` from v0.9.1 — `{ status: 'idle' | 'loading' | 'present' | 'absent' | 'error', index?: IconIndex }`.

`<ItemIcon>` rewrite (sketch):
```ts
const native = useManifestState(gameId);
const index = useIconIndex();
const routedManifests = useRoutedManifests(gameId, category, id, index);
const resolved = resolveIcon({ gameId, category, id }, { ...routedManifests, [gameId]: native }, index.value);
```

`useRoutedManifests` lazy-loads ONLY the manifests the resolver might need (intersection of `index.items["<cat>/<id>"]` and `ROUTING_PRIORITY`), preventing thrash for games the user hasn't activated.

`useGameHasIcons(gameId)` semantics expand: a game returns `true` if it has native icons OR if all its catalog items are covered via routing. Implementation: precompute coverage from `index` + alias map at startup, memoize.

Three PRs (not two) because folding the audit doc into PR 1 makes the diff noisy and folding it into PR 3 means the audit can't be reviewed independently of release-prep churn.

## 9. Open questions

1. **Alias map authority.** Does the alias map live in `iconRouting.ts` (compiled in) or in `index.json` (data-only)? This plan picks data-only (§1 schema includes `aliases`). Alternative: keep aliases in TS so they're typed and grep-able. Decision should hold before PR 1.
2. **Per-category routing priority overrides.** Worth it? Current plan says no (single global priority). If hand-drawn art lands ACGCN-first for fish but ACNH-first for art (because ACGCN art is just paintings that look identical to ACNH paintings), we may want overrides. Defer until concrete need.
3. **Routing observability.** Should `<ItemIcon>` set a `data-icon-source="native|routed:acgcn|placeholder"` attribute for devtools inspection? Useful for debugging coverage holes; trivial to add. Lean yes.
4. **Index generation hook.** Tied to pre-commit, pre-build, or manual `npm run generate:icons`? v0.9.1 scrape pipeline already has hooks — confirm which one it uses and align.
5. **Stale-index detection at runtime.** If `index.json` is older than any per-game manifest, surface a console warning in dev mode? Cheap insurance against forgetting to regenerate. Lean yes for dev-only path.
6. **Fossil routing revisit trigger.** What concrete signal flips fossils from §4 Option A to B/C? Probably "user-visible coverage gap > N items in a shipped beta". Not a v0.9.2 decision but worth recording.
