# v0.9.2-beta ‚Äî Flat Icon Hierarchy (Scoping Plan)

**Status:** scoping pass; nothing implemented. Supersedes the earlier "cross-game routing" framing on this branch ‚Äî that design solved the wrong problem.

The actual model: **one drawing per item id, period.** Sea bass is sea bass regardless of game. There is no "native ACGCN sea bass" vs. "routed-from-ACGCN sea bass" ‚Äî there's just THE sea bass icon. So the architecture is one flat icon hierarchy, not five per-game hierarchies with a routing layer between them.

```
public/icons/
  fish/<id>.<ext>
  bugs/<id>.<ext>
  fossils/<id>.<ext>
  art/<id>.<ext>
  sea_creatures/<id>.<ext>
  manifest.json
```

v0.9.1 shipped `public/icons/<gameId>/<category>/<id>.<ext>` with a per-game manifest. v0.9.2 is a flatten + migrate.

---

## 1. Flat manifest schema

`public/icons/manifest.json`:

```json
{
  "fish":          { "sea-bass": "png", "pale-chub": "png" },
  "bugs":          { "common-butterfly": "png" },
  "fossils":       { "placeholder": "png" },
  "art":           { "famous-painting": "png" },
  "sea_creatures": { "spider-crab": "png" }
}
```

Same shape as the v0.9.1 per-game manifest; one file at the top level instead of one per game. Generated by `scripts/generate-icon-manifest.ts` (renamed/updated from the v0.9.1 version) which scans `public/icons/<category>/`.

Values are file extensions, not full filenames ‚Äî the filename is always `<id>.<ext>`. (v0.9.1 stored full filenames because mixed extensions per game made that simpler; with one flat tree, `<id>.<ext>` is the invariant.)

---

## 2. Resolver ‚Äî trivial

`<ItemIcon>` takes `(category, id)`. `gameId` stays as a prop on the component (callers pass it for non-icon reasons), but the URL builder ignores it.

```
1. canonicalId = RENAME_OVERRIDES[id] ?? id
2. ext = manifest[category]?.[canonicalId]
3. ext  ‚Üí /icons/<category>/<canonicalId>.<ext>
   none ‚Üí null  (render placeholder)
```

No priority chain. No per-game folder. No native-vs-routed.

---

## 3. Migration plan

```
public/icons/acgcn/fish/*    ‚Üí public/icons/fish/*
public/icons/acgcn/bugs/*    ‚Üí public/icons/bugs/*
public/icons/acgcn/fossils/* ‚Üí public/icons/fossils/*
public/icons/acgcn/art/*     ‚Üí public/icons/art/*
```

Use `git mv` to preserve history. Delete `public/icons/acgcn/` and its `manifest.json` after the move. Regenerate the new flat `public/icons/manifest.json` via the updated generator script.

If any other per-game directories exist on disk that aren't yet committed (acww/, accf/, etc.), they merge upward with the same rule ‚Äî first writer wins per id, since the canonical model says they're the same drawing. Conflicts (different bytes for the same id) are an audit smell to flag, not a thing to "resolve" ‚Äî log a warning and keep the first.

The flatten is mechanical and reversible from git history if anything goes wrong.

## 5. Fossil treatment

Single generic placeholder for v1.0 per `roadmap-to-v1.md`. Cleanest implementation under the flat model:

**Drop one `public/icons/fossils/placeholder.png`. In the resolver, special-case `category === 'fossils'` to ignore the input id and resolve to `fossils/placeholder.<ext>`** (looked up in the manifest like any other entry, so swapping the file out is a no-code change).

This is cleaner than putting every fossil id into `RENAME_OVERRIDES` mapping to `placeholder` (which would bloat the map with hundreds of entries and obscure what real renames look like). The category-level special case is one branch in the resolver, easy to remove when per-piece fossil art arrives.

The ACNH per-piece vs. older-games per-skeleton schema drift becomes irrelevant: every fossil id, of either shape, resolves to the same placeholder.

---

## 6. Component changes

- `<ItemIcon>` URL builder drops `gameId`; resolver becomes the ¬ß2 chain. `gameId` prop stays (existing callsites pass it, no churn).
- `useManifestState(gameId)` ‚Üí `useManifestState()` (no arg). Module-level cache becomes a single entry instead of a `Record<GameId, ...>`. Single fetch per session.
- **Drop `useGameHasIcons(gameId)`.** Under the flat model it's no longer a meaningful question ‚Äî every game uses the same manifest, and missing icons render placeholders inline. Callers that gated rendering on it lose the gate; the placeholder behavior is acceptable in those spots (verify during PR (a) ‚Äî likely it was used in `<ItemIcon>` itself or in lists; the simplification is the point).
- No per-item-existence hook to replace it. If a future caller needs "does this specific item have an icon", add `useHasIcon(category, id)` then; not now.

## 7. Audit script

Repurposed, not removed: `scripts/audit-icon-coverage.ts` reads each per-game data catalog (`public/data/<gameId>/*.json`), applies `RENAME_OVERRIDES`, and reports per-game gap lists against the single flat manifest. Output: `docs/v0.9.2-icon-coverage-audit.md` with per-game uncovered items.

The audit no longer drives a routing layer ‚Äî it drives the **"what to draw next"** list for v0.9.4-v0.9.7 scoping.

## 9. Open questions

1. **Manifest values: extension string vs. full filename.** Plan picks `<ext>` because filenames are now `<id>.<ext>` by invariant. If any future case wants a different stem (rare), revert to full filename. Confirm before PR (a).
2. **Fossil placeholder filename.** `placeholder.png`? `_placeholder.png`? Whatever's chosen, document it so the manifest entry and the resolver special case agree.
3. **Conflict-on-flatten policy.** First-writer-wins with a logged warning is the proposal. If any conflict appears during the actual `git mv` pass, surface it in the PR body for explicit decision.
4. **`useGameHasIcons` removal blast radius.** Need to grep callsites in PR (a) and confirm none of them genuinely needed game-level gating. Expected: zero real uses; if there's one, address inline.
5. **Manifest staleness in dev.** Worth a console warning if `manifest.json` is older than any file in `public/icons/<cat>/`? Cheap, dev-only. Lean yes.
6. **Audit script extension cost.** Adding a second category (e.g. furniture in a future release) means one entry in the script's category list. Document that so future-self isn't relearning.
